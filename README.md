# ระบบเช็คยอด COD & บัตรเครดิต — Perfect Group Intertrading

เว็บแอปสำหรับเทียบยอดโอน COD จากขนส่ง (Best / J&T / Kerry) และยอดหักบัตรเครดิต (PaySolution)
กับ Statement ธนาคาร SCB เพื่อหารายการที่โอนช้าผิดปกติหรือยังไม่เข้า พร้อมเก็บประวัติย้อนหลังผ่าน Firebase

## วิธี Deploy ผ่าน GitHub Pages

1. สร้าง repo ใหม่ในบัญชี GitHub (pawineeperfect2540-a11y) เช่น `cod-recon`
2. อัปโหลดไฟล์ทั้งหมดในโฟลเดอร์นี้ (`index.html`, `app.js`, `firebase-config.js`) ขึ้น repo
3. ไปที่ Settings → Pages → Source เลือก branch `main` โฟลเดอร์ `/root` → Save
4. รอสักครู่ เว็บจะขึ้นที่ `https://pawineeperfect2540-a11y.github.io/cod-recon/`

(หรือจะ deploy ผ่าน Vercel เหมือนระบบ deposit-form-app เดิมก็ได้ — เชื่อม repo แล้ว deploy อัตโนมัติ)

## วิธีตั้งค่า Firebase (สำหรับเก็บประวัติย้อนหลัง)

ถ้าไม่ตั้งค่าขั้นตอนนี้ เว็บจะยังใช้เช็คยอดได้ปกติ แต่จะไม่บันทึกประวัติย้อนหลัง

1. เข้า https://console.firebase.google.com
2. สร้างโปรเจกต์ใหม่ หรือใช้โปรเจกต์เดียวกับระบบ deposit-form-app เดิม (แนะนำ ถ้ามีอยู่แล้ว จะได้ไม่ต้องตั้งสิทธิ์ใหม่)
3. เปิดใช้งาน **Firestore Database** (Build → Firestore Database → Create database → เลือกโหมด production)
4. ไปที่ Project Settings (รูปเฟือง) → General → Your apps → ถ้ายังไม่มี web app ให้กด "Add app" เลือก Web
5. คัดลอกค่า `firebaseConfig` มาใส่ในไฟล์ `firebase-config.js` แทนค่าว่างทั้งหมด แล้วเปลี่ยน `FIREBASE_ENABLED` เป็น `true`
6. ตั้งค่า Firestore Security Rules ให้เหมาะสม (เช่น จำกัดเฉพาะคนในทีมที่เข้าถึงได้ ผ่าน Firebase Auth หรือจำกัด IP — แนะนำให้ปรึกษาทีม IT ถ้าต้องการความปลอดภัยเพิ่มเติม เพราะค่าเริ่มต้นของ Firestore เปิดกว้างสำหรับการทดสอบเท่านั้น)

## วิธีใช้งาน

1. แท็บ **อัปโหลดข้อมูล**: อัปโหลดไฟล์ COD จาก Best / J&T / Kerry (อัปโหลดได้หลายไฟล์พร้อมกัน), อัปโหลด Statement ธนาคาร SCB (อัปโหลดได้หลายไฟล์ ระบบแยกตามเลขบัญชีให้อัตโนมัติ), และไฟล์ orderlist.xlsx จาก PaySolution
2. กด "ประมวลผลและจับคู่ยอด"
3. ดูผลในแท็บ **ผลการตรวจสอบ** — กรองดูเฉพาะ "โอนช้า" หรือ "ยังไม่พบ" เพื่อตามขนส่งได้ทันที
4. Export เป็น Excel เพื่อส่งสรุปผู้บริหาร
5. แท็บ **ประวัติย้อนหลัง**: เลือกเดือนเพื่อดูข้อมูลที่เคยบันทึกไว้ (ต้องตั้งค่า Firebase ก่อน)
6. แท็บ **ตั้งค่า**: แก้ไขเลขบัญชีที่ผูกกับแต่ละขนส่ง และจำนวนวันที่ถือว่า "ปกติ" (ค่าเริ่มต้น: เข้าไม่เกิน 1 วันหลังวันที่ขนส่งแจ้งว่าจะโอน = ปกติ, เกินกว่านั้น = ช้า)

## หมายเหตุสำคัญเกี่ยวกับข้อมูลตัวอย่างที่ใช้ทดสอบ

- **J&T**: ยังไม่มีไฟล์ตัวอย่างจริง ระบบใช้โครงสร้างคอลัมน์แบบเดียวกับ Best เป็นค่าเริ่มต้นไปก่อน (Waybill Number, Sign DT, Remittance DT, COD Amount) — เมื่อมีไฟล์จริงจาก J&T ควรทดสอบและแจ้งปรับ parser ให้ตรงหากโครงสร้างต่างออกไป
- **บัตรเครดิต (PaySolution)**: จากการทดสอบกับไฟล์ orderlist.xlsx ตัวอย่าง พบว่ายอด Net ของแต่ละออเดอร์ไม่ได้เข้าบัญชีตรงตามวันที่ในไฟล์เป๊ะๆ — PaySolution น่าจะรวมยอดหลายวันเป็นรอบโอนของตัวเอง (เหมือนขนส่ง) ไม่ใช่โอนแยกเป็นรายวันตามวันที่ลูกค้าจ่าย ถ้า PaySolution มีรายงาน "สรุปยอดโอน" (remittance/settlement report) แยกต่างหาก จะช่วยให้จับคู่แม่นยำกว่าการใช้ orderlist.xlsx เพียงอย่างเดียว — แนะนำให้ลองขอไฟล์ประเภทนี้จาก PaySolution เพื่อทดสอบเพิ่มเติม
- ตรรกะการจับคู่ปัจจุบัน: รวมยอดพัสดุ/ออเดอร์ตามวันที่ขนส่ง/PaySolution แจ้งว่าจะโอน (batch) แล้วหาเงินฝากในบัญชีที่มียอดตรงกัน (คลาดเคลื่อนได้ไม่เกิน 1 บาท) ภายในช่วง 1 วันก่อน ถึง 30 วันหลังวันที่แจ้ง — ถ้าพบว่ายอด "ไม่ตรงเป๊ะ" บ่อยๆ ในทางปฏิบัติ (เช่น มีการหักค่าใช้จ่ายอื่นปนอยู่) อาจต้องปรับ tolerance หรือ logic เพิ่มเติม
